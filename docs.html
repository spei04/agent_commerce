Agent payments you can govern.

Agent Commerce is a governance and execution layer for agent-initiated spend. Agents transact through programmable wallets. Organizations get approvals, audit trails, and webhook-driven state — designed like real payment rails, but simulated for the demo.

Human Auth

Authorization: Bearer <jwt>

Agent Auth

X-Wallet-Key: wk_<key_id>.<secret>

Demo

Open /demo to see approvals + settlement + policy trace live.

Overview

This is a multi-tenant payments control plane. Each agent is scoped to a wallet, and each wallet is scoped to an organization. Every purchase attempt is evaluated by policy, recorded for audit, and executed with card-like semantics (authorize → capture).

Design principle: approvals are guaranteed. If a human approves a request, the funds were reserved at request-time, so the approval can’t fail due to concurrent spend.

Quickstart

For the demo, the server seeds a demo org + user on startup.

curl -s -X POST http://localhost:8000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"demo@agentcommerce.local","password":"demo-password"}'

Then open: http://localhost:8000/demo

Local DB setup: the intended Iteration 2 dev path is Postgres + Alembic. If Docker isn’t available, you can run on SQLite by setting DATABASE_URL (see README).

Tenancy & Identity

Humans

Humans (finance ops / procurement approvers) authenticate with JWTs.

Authorization: Bearer <jwt>

Agents

Agents authenticate with wallet-scoped API keys. Each key maps to exactly one wallet.

X-Wallet-Key: wk_<key_id>.<secret>
Confused deputy protection: the API infers wallet from the key. Avoid passing wallet IDs in agent requests; the server treats the key as the source of truth for scope.

Agents, Wallets, Keys

An agent can own multiple wallets, distinguished by purpose (procurement, cloud, data, travel, other). A wallet can have multiple API keys for rotation and revocation. 

# Create an agent (human/admin)
curl -s -X POST http://localhost:8000/agents \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"external_agent_id":"procure-bot","name":"Procurement Bot","metadata":{"team":"ops"}}'

# Create a wallet (human/admin)
curl -s -X POST http://localhost:8000/wallets \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"agent_id":"<agent_id>","purpose":"procurement","name":"Procurement Wallet","balance_cents":500000,"auto_approve_limit_cents":5000,"spend_limit_max_cents":30000}'

# Create a wallet API key (human/admin) — returned once
curl -s -X POST http://localhost:8000/wallets/<wallet_id>/keys \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"name":"agent-key-1"}'

Money Model

All money is stored as integer minor units (cents). Wallet invariants:

  • available_cents = balance_cents - reserved_cents
  • reserved_cents increases on holds and decreases on capture/release
  • Reservations are applied atomically to protect against concurrent purchases
Why reserved exists: approvals are only trustworthy if the platform “earmarks” funds at request-time. Otherwise a malicious or buggy agent can race spend between request and approval.

Approvals (Guaranteed)

If a purchase requires approval, the platform creates an approval request and reserves funds immediately. Approvers are approving a snapshot: the exact vendor/product/amount that was requested. 

# List pending approvals (approver/admin)
curl -s "http://localhost:8000/approvals?status=pending" \
  -H "Authorization: Bearer <jwt>"

# Approve exactly what was requested
curl -s -X POST http://localhost:8000/approvals/<approval_id>/approve \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"note":"ok","rail":"card"}'

# Or reject (releases the hold)
curl -s -X POST http://localhost:8000/approvals/<approval_id>/reject \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"note":"no"}'

Payment Rails (Simulated)

Card-like semantics

A successful purchase creates an authorization (hold). Settlement happens asynchronously via capture. The agent sees payment_status=processing until capture occurs.

ACH payout semantics

Procurement-like “invoice approval → payout” flow. Approvers can approve with rail=ach, which creates an ACH payout that settles asynchronously to paid or failed.

Purchase API

Agents call POST /purchase with an idempotency key. Wallet scope is inferred from X-Wallet-Key. 

curl -s -X POST http://localhost:8000/purchase \
  -H "X-Wallet-Key: wk_<key_id>.<secret>" -H "Content-Type: application/json" \
  -d '{"product_id":"<product_id>","quantity":1,"intent":"buy packaging","idempotency_key":"<uuid>","rail":"card"}'
Best practice: treat pending_approval as a terminal state for the agent. Humans continue the workflow. Agents should wait for webhooks or poll via an orchestrator.

Webhooks

Webhooks are delivered at-least-once via an outbox. Receivers must dedupe by event.id. Signatures are per-wallet. 

# Configure webhook URL + rotate signing secret (human/admin)
curl -s -X POST http://localhost:8000/wallets/<wallet_id>/webhook \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"webhook_url":"https://example.com/webhook","rotate_secret":true}'
# Receiver pseudocode
raw = request.body
sig = request.headers["X-Agent-Commerce-Signature"]  # v1=<hex>
expected = "v1=" + hex(hmac_sha256(webhook_signing_secret, raw))
assert constant_time_equal(sig, expected)
event = json(raw)
dedupe(event["id"])
Delivery model: at-least-once means duplicates happen. Build receivers to be idempotent.

Replay

Replay a specific webhook event id (useful for debugging and trust). Requires approver role.

curl -s -X POST http://localhost:8000/webhooks/replay \
  -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" \
  -d '{"event_id":"<event_id>"}'

Worker / Tick

The demo UI uses POST /simulate/tick. The production-friendly direction is a dedicated worker process draining outbox + settling rails. 

# Manual tick (human/admin)
curl -s -X POST http://localhost:8000/simulate/tick \
  -H "Authorization: Bearer <jwt>"

Roles & RBAC

  • admin: create agents, wallets, keys; seed demo; view all org data
  • approver: approve/reject approval requests; replay webhooks
  • auditor: read-only access to org transactions and payment records

Idempotency

Always send idempotency_key on agent calls. On timeouts or network failures, retry the same request with the same key. 

# Pseudocode
key = uuid()
resp = POST /purchase { idempotency_key: key, ... }
if timeout:
  retry POST /purchase { idempotency_key: key, ... }

Errors

  • 401: missing/invalid JWT or wallet key
  • 403: role is insufficient (RBAC)
  • 400: policy blocked or insufficient balance under concurrency
  • 409: invariant conflict (example: missing hold for approval)

FAQ

Why reserve funds on approval creation?

Because “approved” must mean “will execute.” Reservations eliminate the race where an agent spends funds before the human approves.

Are webhooks exactly-once?

No. They’re at-least-once. Deduplicate by event id.

Can an agent own multiple wallets?

Yes. Wallets are distinguished by purpose and scoped under an org.

Overview (Finance Approvers)

Agent Commerce routes certain agent purchases to a human approval queue. Approvals are safe: when a request appears in the queue, the funds are already reserved in the agent’s wallet.

What “Approve” means: you are approving the exact request snapshot (vendor/product/amount). The platform will execute the payment flow (simulated in this demo) and record a permanent audit trail.

Approval Workflow

1) Review the request

  • Vendor + product
  • Amount + currency
  • Agent intent (why the agent says it needs this spend)
  • Policy trace (which guardrails triggered)

2) Approve or reject

Approving creates an executed transaction. Rejecting releases the reserved funds and logs the decision for audit.

Tip: if you see repeated approvals for the same vendor/intent, ask the team to increase auto-approve thresholds or add a vendor allowlist to reduce approval load.

Guaranteed Approvals

The platform reserves funds when the approval request is created. This prevents “approval races” where an agent spends the balance elsewhere before you approve.

  • available = balance - reserved
  • Approval creation increases reserved
  • Approve converts reserved → spent; reject releases reserved

Rails: Card vs ACH

In the long-term product, “rail selection” can map to different vendor payout methods. In this demo, both rails are simulated but preserve realistic semantics.

  • card: fast authorize → capture behavior (agent buys now)
  • ach: slower payout semantics (invoice approval → payout)

Audit & Reporting

Every decision is logged:

  • Blocked purchases (with reasons)
  • Approval requests + reviewer notes
  • Executed transactions + settlement outcomes
Operational model: approvals are for exceptions. The goal is to converge policy so most spend is safe and automatic.

Troubleshooting

  • If approvals aren’t changing state in the demo: confirm you’re logged in (JWT) and the tick/worker is running.
  • If you see duplicates in webhook receivers: dedupe by event id (delivery is at-least-once).
  • If a wallet becomes suspended: it likely hit a velocity rule; an admin can adjust limits or unsuspend in the next iteration.

Roles

  • approver: approve/reject spend requests
  • admin: manage wallets/keys/policies for an org
  • auditor: read-only visibility for reporting

FAQ (Finance)

Can approvals fail?

No, not due to lack of funds. Funds are reserved at request-time.

What exactly am I approving?

The request snapshot: vendor/product/amount and the agent’s stated intent.

Why do I see “processing”?

Because settlement is asynchronous (even in the simulated demo). It transitions to succeeded/failed.
