AGENT COMMERCE — INTEGRATION CHEATSHEET

Auth:
- Admin endpoints: X-Admin-Key (default demo-admin)
- Agent endpoints: X-Wallet-Key (wallet_key returned ONCE by POST /wallets)

Core flow:
1) POST /resolve (optional discovery)
2) POST /purchase with idempotency_key (required for safe retries)
3) If status=pending_approval: human/admin approves via POST /approvals/{id}/approve

Reliability:
- Retry by re-sending the same idempotency_key (prevents double-charging / duplicate approvals)
- Payments settle async: payment_status = processing → succeeded|failed
- Holds reserve funds: reserved_balance reduces available_balance

Webhooks:
- Delivery is at-least-once (duplicates possible)
- Verify: X-Agent-Commerce-Signature = v1=<hex(hmac_sha256(wallet_key, request_body))>
- Event types: approval_required, payment_succeeded, payment_failed
A
Create Wallet (Admin)
X-Admin-Key required. Response returns wallet_key once.
{
curl -s -X POST http://localhost:8000/wallets \
  -H "X-Admin-Key: demo-admin" -H "Content-Type: application/json" \
  -d '{"agent_id":"my-agent","name":"My Agent","balance":500,"auto_approve_limit":50,"spend_limit_max":300,"allowed_vendors":[]}'
}
B
Purchase (Agent)
X-Wallet-Key required. Always send idempotency_key.
{
curl -s -X POST http://localhost:8000/purchase \
  -H "X-Wallet-Key: <wallet_key>" -H "Content-Type: application/json" \
  -d '{"wallet_id":"<wallet_id>","product_id":"<product_id>","quantity":1,"intent":"order supplies","idempotency_key":"<uuid>"}'
}
C
Approve (Human/Admin)
Approvals are snapshot decisions: approve exactly what was requested. Settlement happens async.
{
curl -s -X POST http://localhost:8000/approvals/<approval_id>/approve \
  -H "X-Admin-Key: demo-admin" -H "Content-Type: application/json" -d '{}'
}
Payments are simulated async: payment_status transitions processing → succeeded|failed. Wallets expose reserved_balance and available_balance.
Agent Reliability Notes
Always retry safely with idempotency_key. Treat webhooks as at-least-once (duplicates happen).
R
Idempotency
On network errors/timeouts, re-send the same idempotency_key and return the existing transaction/approval.
{
// Pseudocode
key = uuid()
try purchase(key)
catch retry purchase(key)  // same key
}
W
Webhook Signature
Verify X-Agent-Commerce-Signature using HMAC-SHA256 with the wallet key as the secret.
{
signature = "v1=" + hex(hmac_sha256(secret=wallet_key, message=request_body))
}
S
Settlement
A successful purchase returns immediately with payment_status=processing. Poll GET /transactions or rely on webhooks for completion.
Integration Recipes
Short patterns agents can follow without reading the whole API.
1
Purchase With Retry
Generate a UUID once. Reuse it on any retry until you get a definitive response.
{
idempotency_key = uuid()
resp = POST /purchase { ..., idempotency_key }
if network_error: retry POST /purchase with SAME idempotency_key
}
2
Handle Pending Approval
Treat pending_approval as a final state for the agent; a human continues the workflow.
{
resp = POST /purchase ...
if resp.status == "pending_approval":
  approval_id = resp.approval_request_id
  notify human + wait for webhook(payment_succeeded|failed) or poll /transactions
}
3
Wait For Settlement
Both auto-approved and approved-by-human settle asynchronously. Poll or use webhooks.
{
txn = POST /purchase ...
while txn.payment_status == "processing":
  sleep(0.5s)
  txn = GET /transactions?wallet_id=... (find txn.id)
}
4
Verify Webhook
Webhooks are at-least-once. Verify signature, then dedupe by event id.
{
raw = request_body
sig = header["X-Agent-Commerce-Signature"]
assert sig == "v1=" + hex(hmac_sha256(wallet_key, raw))
event = json(raw)
dedupe(event.id)
}
